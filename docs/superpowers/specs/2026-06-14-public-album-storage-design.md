# 公开相册与 Storage 基础设计

## 目标

流程 5 拆分为两个子流程：

- 流程 5A：Supabase Storage 基础、公开相册管理、真实公开相册和首页相册照片；
- 流程 5B：头像、Favicon、作品封面与截图、收藏封面接入上传能力。

本规格仅覆盖流程 5A。目标是将当前空白拍立得演示升级为由唯一管理员维护的极简公开照片分享系统，同时建立可供流程 5B 延续使用的安全 Storage 基础。

流程 5A 不建设传统相册分类或照片档案。照片没有公开标题、描述、拍摄日期、地点或标签；核心体验是直接分享照片。

## 前置条件

- Pull Request #4 必须先合并；
- 流程 5A 从最新 `main` 创建独立 worktree 和分支；
- 以真实 Supabase 云项目作为最终迁移与验收环境；
- Docker Desktop 可以作为可选工具，但不是实现或验收的必需步骤；
- 不安装新的图片处理依赖，浏览器优先使用 Canvas 与标准 Web APIs。

## 范围

### 本流程包含

- 创建共用公开 Storage 桶 `public-media`；
- 创建服务器专用 `photos` 数据表；
- 创建受保护的照片上传、替换和删除接口；
- 创建照片后台列表、批量上传、编辑和回收站；
- 将 `/album` 替换为真实拍立得堆叠相册；
- 将首页相册模块接入真实公开照片；
- 建立 Storage、数据库和文件生命周期的一致性规则；
- 增加自动化测试、云端权限验收和外部浏览器验收。

### 本流程不包含

- 私人照片、私有 Storage 桶或签名 URL；
- 头像、Favicon、作品封面、作品截图或收藏封面上传；
- 相册分组、年份筛选、标签筛选或照片搜索；
- 公开标题、描述、日期、地点、摄影参数或其他照片元数据；
- 下载按钮、水印、禁止右键或阻止浏览器保存；
- 文件哈希去重、自动清理任务或历史文件版本；
- 批量发布、批量归档、批量删除或拖拽排序；
- 新图片处理依赖或强制本地 Docker Supabase。

## 数据模型

新增 `photos` 表：

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `id` | UUID | 主键；上传路径生成前由服务端创建 |
| `storage_path` | Text | 必填；系统生成的 `public-media` 桶内路径 |
| `original_filename` | Text | 必填；仅供后台识别，不公开展示 |
| `visibility` | Enum | `draft`、`public`、`archived` |
| `sort_order` | Integer | 非负整数，数值越小越靠前 |
| `deleted_at` | Timestamptz / Null | 有值时位于回收站 |
| `created_at` | Timestamptz | 创建时间 |
| `updated_at` | Timestamptz | 最后修改时间 |

规则：

- 上传成功后自动创建草稿；
- 公开查询只返回 `visibility = public` 且 `deleted_at is null` 的记录；
- 公开排序先按 `sort_order` 升序，再按 `created_at` 降序；
- 从回收站恢复后统一变为草稿；
- 原始文件名不得进入公开页面数据；
- `photos` 启用 RLS，并撤销 `anon` 与普通 `authenticated` 角色的直接表权限；
- 浏览器不直接查询或写入 `photos`。

新增最小的服务器专用 `storage_cleanup_tasks` 表，用于持久化无法立即删除的 Storage 对象：

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `id` | UUID | 主键 |
| `bucket_id` | Text | 本流程固定为 `public-media` |
| `object_path` | Text | 待清理对象路径，必须唯一 |
| `reason` | Text | 安全的失败来源标识，不保存内部堆栈 |
| `last_error` | Text / Null | 面向管理员的安全错误摘要 |
| `created_at` | Timestamptz | 创建时间 |
| `updated_at` | Timestamptz | 最后重试时间 |

该表不是通用媒体库，不关联公开页面，也不自动执行任务。它仅保证创建回滚或文件替换时的清理失败可以在后台被追踪和手动重试。该表同样启用 RLS，并撤销浏览器角色的直接权限。

## Storage 与安全边界

创建公开存储桶 `public-media`。流程 5A 使用以下路径：

```text
album/{photo-id}/{generated-name}.webp
```

路径与文件名均由服务端生成，不使用管理员提供的路径，也不保留原始文件名作为对象名。

权限规则：

- `public-media` 允许公开读取；
- `anon` 和普通 `authenticated` 浏览器角色不得上传、替换或删除；
- Storage 策略不向浏览器角色创建 `insert`、`update` 或 `delete` 写策略；
- 所有写操作经过受保护的 Next.js Route Handler；
- 每次写操作在使用 Secret Key 前重新验证当前管理员；
- 服务端必须重新验证文件类型、大小、目标照片和生成路径；
- Secret Key、管理员 ID、会话值和内部错误不得进入客户端响应、日志或 Git；
- 草稿和归档文件仍可能通过已知公开 URL 直接访问，但不会出现在公开页面。

公开读取是产品选择，不代表允许浏览器写入。

## 上传与图片处理

管理员每次最多选择 `10` 张图片，最多同时处理和上传 `2` 张。

接受输入：

- JPEG；
- PNG；
- WebP；
- 单张原文件最大 `10 MB`。

浏览器处理：

- 将最长边限制为 `2560px`；
- 统一输出 WebP，质量约 `85%`；
- 透明 PNG 转换后保留透明度；
- 通过重新编码清除 EXIF 与 GPS 信息；
- 每张照片独立显示处理中、上传中、成功和失败状态。

安全边界：

- 浏览器处理结果不被服务端信任；
- Route Handler 重新验证身份、MIME 类型、处理后文件大小和请求参数；
- Route Handler 拒绝超过 `10 MB` 的处理后文件；
- 处理或上传失败只影响当前文件；
- 成功项保留为草稿，失败项可以单独重试。

创建流程：

1. 浏览器处理单张图片；
2. 服务端验证管理员并生成照片 ID 与 Storage 路径；
3. 服务端上传文件到 Storage；
4. Storage 上传成功后创建草稿记录；
5. 数据库记录创建失败时，立即尝试删除已上传文件；
6. 清理失败时创建 `storage_cleanup_tasks` 记录并报告给管理员，不伪装成功。

## 后台管理

新增路由：

| 路由 | 职责 |
| --- | --- |
| `/admin/photos` | 照片网格列表与状态筛选 |
| `/admin/photos/new` | 最多十张照片的批量上传 |
| `/admin/photos/[id]/edit` | 单张照片预览、状态、排序和替换 |
| `/admin/photos/trash` | 回收站、恢复和永久删除 |
| `/admin/photos/cleanup` | 查看并手动重试待清理 Storage 对象 |

后台列表使用图片网格，显示：

- 照片缩略图；
- 原始文件名；
- 草稿、公开或归档状态；
- 排序值；
- 进入编辑和回收站操作。

后台不显示或编辑标题、描述、日期、地点或标签。

编辑页显示：

- 完整照片预览；
- 拍立得卡片实时预览；
- 原始文件名；
- 状态；
- 数字排序值；
- 替换照片文件操作。

流程 5A 支持批量上传，但编辑、发布、归档、回收、恢复与永久删除均逐张执行。

## 文件替换、删除与一致性

### 替换照片

1. 先上传新文件到同一照片目录下的新生成路径；
2. 更新数据库记录指向新文件；
3. 数据库更新成功后删除旧文件；
4. 旧文件删除失败时保留新记录，并向管理员报告待清理路径；
5. 旧文件删除失败时创建 `storage_cleanup_tasks` 记录；
6. 后台提供针对待清理任务的手动重试入口，清理成功后删除任务记录。

### 回收站

- 移入回收站时保留 Storage 文件；
- 恢复后状态统一变为草稿；
- 永久删除时先删除当前 Storage 文件；
- Storage 删除成功后才删除数据库记录；
- 删除失败时保留数据库记录并提示重试，避免产生无法追踪的孤立文件。

流程 5A 不建设自动定时清理任务。无法立即清理的失败必须保存为 `storage_cleanup_tasks`，并由管理员手动重试。

## 公开 `/album` 体验

公开相册使用纯拍立得堆叠，不使用摄影墙或传统网格。

分组：

- 每组最多 `12` 张公开照片；
- 当前组第一张默认位于顶部；
- 其余照片使用基于照片 ID 的稳定偏移与旋转；
- 支持上一组和下一组；
- 当前组号写入 URL 查询参数，例如 `/album?group=2`；
- 非法、负数或超范围组号回到有效组；
- 没有公开照片时显示克制空状态，不显示演示照片。

交互：

- 点击非顶部照片将其置顶；
- 再次点击顶部照片打开灯箱；
- 切换组时，当前组淡出并轻微滑走，新组重新堆叠；
- 卡片使用统一裁切预览；
- 灯箱显示完整比例照片；
- 不显示标题、描述或其他元数据。

灯箱：

- 可以浏览全部公开照片，不受当前 12 张分组限制；
- 支持上一张、下一张和关闭按钮；
- 点击背景或按 `Escape` 关闭；
- 左右方向键切换照片；
- 图片加载失败时显示受控错误提示。

移动端：

- 继续使用拍立得堆叠；
- 减少偏移与旋转；
- 控件不得超出屏幕；
- 不允许页面级横向滚动。

可访问性：

- 每张拍立得可以键盘聚焦；
- `Enter` 和 `Space` 执行与点击一致的置顶或打开行为；
- 灯箱打开时管理焦点并支持方向键与 `Escape`；
- 关闭灯箱后将焦点返回触发照片。

## 首页相册模块

首页相册模块接入真实公开照片：

- 每次服务端渲染随机选择最多三张公开照片；
- 使用拍立得视觉展示；
- 点击模块进入 `/album`；
- 没有公开照片时显示克制空状态；
- 查询失败时显示受控错误状态，不影响首页其他模块；
- 不再展示空白演示拍立得。

首页随机查询由 Next.js 服务端执行，不向浏览器开放数据库访问。

## 错误处理

### 上传与管理

- 每张文件独立报告格式、大小、处理和上传错误；
- 一张失败不取消其他成功上传；
- Storage 上传、数据库创建、替换、文件删除或数据库删除失败均不得伪装成功；
- 清理失败时保留可重试信息；
- 客户端只接收面向管理员的安全错误消息，不接收内部堆栈或密钥。

### 公开页面

- 查询失败时显示受控错误状态；
- 单张图片加载失败时显示拍立得占位背景；
- 灯箱图片加载失败时显示明确提示；
- 其他站点页面和首页模块继续工作。

## 迁移与权限

版本化 SQL 迁移负责：

- 创建照片状态枚举、`photos` 表、约束、索引和更新时间触发器；
- 创建服务器专用 `storage_cleanup_tasks` 表；
- 启用 RLS；
- 撤销 `anon` 与普通 `authenticated` 对照片表和清理任务表的直接权限；
- 向服务器角色授予必要权限；
- 创建 `public-media` 存储桶；
- 配置公开读取与禁止浏览器角色写入的 Storage 策略。

真实云端最终验收必须确认：

- `photos` 表存在并启用 RLS；
- `storage_cleanup_tasks` 表存在并启用 RLS；
- `anon` 与普通 `authenticated` 对两张表均无直接表权限；
- `public-media` 可以公开读取；
- 浏览器角色不能直接上传、替换或删除 Storage 对象；
- 未登录用户不能访问管理页面或上传接口；
- 管理员可以完成上传、替换、发布、归档、回收、恢复和永久删除；
- 草稿、归档和回收站照片不会出现在公开相册或首页相册模块；
- 永久删除后当前 Storage 文件和数据库记录均不存在。
- 清理失败任务可以被管理员查看、手动重试并在成功后移除。

## 实施边界总结

- 流程 5A 建立专用 `photos` 领域与专用上传接口；
- 不提前建设通用媒体资产表或完整媒体库；
- 上传能力先服务相册，流程 5B 再根据真实复用点提取共享能力；
- `public-media` 从一开始作为流程 5A 与流程 5B 共用的公共存储桶；
- 所有图片写操作保持管理员服务端验证边界；
- 开始实现前必须先合并 Pull Request #4。
