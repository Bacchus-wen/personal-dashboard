# Theodore 个人工作台功能基础设计

## 目标

将当前以静态展示为主的 Next.js 网站升级为一个由站长本人真实使用的个人工作台和公开作品集。网站需要同时支持公开内容与私人数据，并在尽量降低长期维护成本的前提下，建立可靠的认证、授权、数据存储和恢复能力。

第一阶段采用渐进式开发。先完成认证与安全基础，再依次实现近日规划、网站设置和首页布局。每一步都必须独立可验证，安全基础没有通过验证前，不进入内容管理功能开发。

## 网站定位与信息架构

网站定位为：

> 个人工作台 + 网站作品集 + 内容与优秀项目收藏库

规划中的公开页面如下：

| 路由 | 页面 | 用途 |
| --- | --- | --- |
| `/` | 首页工作台 | 展示个人入口、临期规划及各模块摘要 |
| `/plans` | 近日规划 | 展示主动公开的近期规划 |
| `/works` | 我的作品 | 展示网站作品，并跳转至作品网站 |
| `/collections` | 内容收藏 | 展示文章和视频收藏 |
| `/projects` | 优秀项目 | 展示值得关注的 GitHub 项目 |
| `/album` | 相册 | 展示主动公开的照片 |
| `/about` | 关于网站 | 说明网站用途、结构和技术信息 |
| `/resume` | 个人简历 | 展示内部简历并提供 PDF 下载 |

个人简历不占用主要导航位置。首页问候卡片作为简历入口：默认显示右下角小折角，悬浮或键盘聚焦时折角扩大，同时出现低透明度的 Orb 鼠标跟随光效和“查看简历”提示，点击进入 `/resume`。

## 技术架构

采用托管服务减少服务器运维：

- Next.js App Router：公开网站、管理后台、服务端路由保护和业务逻辑。
- Vercel：部署、环境变量和部署历史。
- Supabase Auth：单管理员登录和 MFA。
- Supabase PostgreSQL：内容、设置、布局和操作日志。
- Supabase Storage：公开媒体、私人媒体和简历 PDF。
- GitHub：代码版本记录。

不采用浏览器导入 GitHub App PEM 私钥的认证方式。任何高权限密钥只能存在于服务端环境变量中，不能发送给浏览器。

## 认证与授权

### 管理员模型

- 系统仅支持一个管理员。
- 不提供公开注册入口。
- 管理员使用邮箱和强密码登录。
- 管理后台强制要求验证器 MFA，只有 `aal2` 会话能够执行管理操作。
- 数据库保存固定管理员用户 ID，不能仅通过邮箱判断管理员身份。
- `/admin` 及其子路由必须在服务端验证会话和管理员身份。
- 未登录、未完成 MFA 或不是指定管理员的请求必须被拒绝。

### 登录流程

1. 管理员访问 `/admin/login`。
2. 输入邮箱和密码。
3. 完成 TOTP 验证器 MFA。
4. 服务端确认当前用户 ID 与配置的管理员 ID 一致。
5. 进入 `/admin`。

### 数据权限

所有暴露给应用的数据表必须启用 RLS：

| 操作 | 普通访客 | 管理员 `aal2` |
| --- | --- | --- |
| 查看已发布公开内容 | 允许 | 允许 |
| 查看草稿和私人内容 | 禁止 | 允许 |
| 新增、修改、归档、删除内容 | 禁止 | 允许 |
| 修改网站设置和首页布局 | 禁止 | 允许 |
| 查看操作日志 | 禁止 | 允许 |

前端隐藏按钮不构成权限控制。数据库 RLS 和服务端身份验证是最终授权边界。

## 数据模型

所有可发布内容统一使用以下可见性状态：

- `draft`：草稿，仅管理员可见。
- `published`：公开发布，访客可见。
- `private`：私人内容，仅管理员可见。
- `archived`：归档内容，不在正常页面显示。

第一阶段规划的数据表：

| 表 | 职责 |
| --- | --- |
| `profiles` | 个人资料和简历入口信息 |
| `plans` | 近日规划 |
| `site_settings` | 网站基础配置和功能开关 |
| `social_links` | 社交链接及排序 |
| `home_layout` | 首页桌面布局配置 |
| `audit_logs` | 管理操作记录 |

后续扩展表：

| 表 | 职责 |
| --- | --- |
| `works` | 我的作品 |
| `collections` | 文章和视频收藏 |
| `featured_projects` | 优秀 GitHub 项目 |
| `photos` | 相册照片元数据 |

## 媒体存储

Supabase Storage 使用两个存储桶：

- `public-media`：头像、作品封面、公开相册和网站装饰图片。访客可读，管理员可写。
- `private-media`：私人照片、未发布素材和受保护文件。仅管理员可访问，通过短期签名 URL 读取。

上传必须限制文件类型、大小和尺寸。照片上传流程需要清除 EXIF 定位信息。数据库备份不等同于文件备份，重要媒体必须单独备份。

## 第一阶段开发范围

### 步骤一：认证与安全基础

当前首先执行这一部分：

- 配置 Supabase 客户端与服务端连接。
- 创建单管理员登录页面。
- 创建 TOTP MFA 注册、验证和恢复引导。
- 保护 `/admin` 路由。
- 建立管理员身份检查。
- 建立基础 RLS 策略和权限测试。
- 建立登录、退出、无权限和会话失效流程。
- 编写环境变量和账户恢复说明。

验收条件：

- 未登录用户无法访问后台。
- 普通认证用户无法获得管理员权限。
- 未达到 `aal2` 的管理员无法读取或修改受保护数据。
- 浏览器代码和网络请求中不存在服务端高权限密钥。
- 权限测试能够证明访客、普通认证用户和管理员的访问边界。

### 步骤二：近日规划闭环

- 规划字段：标题、说明、任务状态、优先级、进度、目标日期、相关链接、可见性状态、创建时间和更新时间。
- 支持新增、编辑、归档和移入回收站。
- 新规划默认 `private`，只有管理员主动设置为 `published` 后才对访客可见。
- `/plans` 只展示已发布规划。
- 首页显示距离截止日期最近且已公开的未完成规划。
- 首页临期规划提醒卡片替换当前演示音乐播放器位置。

### 步骤三：网站设置与首页布局

网站设置支持：

- 基础信息。
- 社交链接新增、编辑、删除和排序。
- 首页模块启用或隐藏。
- 首页桌面布局预览、拖拽、保存和恢复默认。

首页拖拽布局限制：

- 卡片吸附到网格。
- 卡片不能拖出画布或互相遮挡。
- 核心卡片不能删除。
- 第一阶段不允许任意修改卡片宽高或手动输入偏移值。
- 移动端继续使用程序定义的固定顺序，不读取桌面拖拽坐标。

后台保留“主题修改”入口，但第一阶段只展示当前默认主题和后续开放说明，不允许任意修改颜色。

## 备份与恢复

- 代码由 GitHub 保存。
- Vercel 保留部署历史。
- 数据库使用 Supabase 托管备份，并提供数据导出流程。
- 重要照片和文件必须定期复制到独立存储位置。
- 删除内容先进入回收站，保留一段恢复期后再永久删除。
- 文档必须记录管理员账户、MFA 恢复、密钥轮换和数据恢复流程，但不能记录真实秘密。

## 协作与 Skills 使用规范

- 每个开发任务开始时，必须向用户说明计划使用的 skills 及其用途。
- 每个开发任务完成时，必须列出实际使用过的 skills、验证命令和验证结果。
- 功能设计使用 `brainstorming`，批准后使用 `writing-plans` 制定实施计划。
- 实施阶段根据任务选择 `test-driven-development`、`systematic-debugging`、`frontend-design`、`verification-before-completion`、`requesting-code-review` 等适用 skills。
- 不为了使用 skill 而使用 skill；每次选择必须与任务风险和目标相关。

## 非目标

第一阶段不实现：

- 多管理员和角色管理。
- 公开用户注册。
- 评论、点赞和社交功能。
- 任意颜色编辑器。
- 任意卡片尺寸和自由重叠布局。
- 浏览器 PEM 私钥认证。
- 自建服务器或自建认证系统。

## 安全验收原则

- 默认拒绝，按需开放。
- 私人内容默认不公开。
- 所有高权限操作必须由服务端和数据库共同验证。
- 任何秘密不得进入 Git、公开前端变量或浏览器存储。
- 每个功能上线前必须测试未登录、低权限和管理员三种身份。
- 安全配置无法被自动验证时，不得仅凭界面行为宣称安全。
