# Theodore 个人工作台功能基础设计

## 目标

将当前静态展示网站升级为由站长本人真实使用的个人工作台和公开作品集。网站同时包含公开内容与私人数据，因此需要可靠的管理员登录、服务端权限检查、私有文件存储和备份能力，但不采用超出个人网站维护能力的企业级权限体系。

第一阶段采用渐进式开发：先完成容易理解、能够真实验证的认证基础，再实现近日规划、网站设置和首页布局。

## 网站定位与信息架构

网站定位为：

> 个人工作台 + 网站作品集 + 内容与优秀项目收藏库

| 路由 | 页面 | 用途 |
| --- | --- | --- |
| `/` | 首页工作台 | 展示个人入口、临期规划及模块摘要 |
| `/plans` | 近日规划 | 展示主动公开的近期规划 |
| `/works` | 我的作品 | 展示网站作品并跳转至作品网站 |
| `/collections` | 内容收藏 | 展示文章和视频收藏 |
| `/projects` | 优秀项目 | 展示值得关注的 GitHub 项目 |
| `/album` | 相册 | 展示主动公开的照片 |
| `/about` | 关于网站 | 说明网站用途、结构和技术信息 |
| `/resume` | 个人简历 | 展示内部简历并提供 PDF 下载 |

个人简历不占用主要导航位置。首页问候卡片作为简历入口：默认显示右下角小折角，悬浮或键盘聚焦时折角扩大，同时出现低透明度 Orb 鼠标跟随光效和“查看简历”提示，点击进入 `/resume`。

## 安全方案

采用适合单管理员作品集网站的中等复杂度方案：

- Supabase Auth：管理邮箱密码、登录会话和密码恢复。
- Next.js 服务端：验证管理员身份，并作为所有数据库和文件写操作的唯一入口。
- Supabase PostgreSQL：保存内容、设置和操作日志。
- Supabase Storage：分别保存公开媒体与私人媒体。
- Vercel：部署网站并保存服务端环境变量。
- GitHub：保存代码和版本历史，仓库优先设为私有。

不采用：

- 浏览器导入 PEM 私钥。
- 自行编写密码保存与认证系统。
- 强制 TOTP MFA、`aal2` 权限体系。
- Docker 本地 Supabase 环境。
- pgTAP 数据库测试。
- 多管理员和复杂角色权限。

## 管理员认证

### 管理员模型

- 系统只支持一个管理员。
- Supabase Auth 关闭公开注册。
- 管理员使用邮箱和强密码登录。
- 服务端使用私有环境变量 `ADMIN_USER_ID` 判断当前登录用户是否为管理员。
- `/admin` 页面和所有后台写操作必须在服务端重新验证身份。
- 管理 GitHub、Vercel、Supabase、域名和邮箱的外部平台账户必须开启平台提供的两步验证。
- Supabase Auth MFA 作为后续可选升级，不是第一阶段阻塞项。

### 登录流程

1. 管理员访问 `/admin/login`。
2. 输入邮箱和密码。
3. Supabase Auth 验证身份并建立登录会话。
4. Next.js 服务端验证当前用户 ID 是否等于 `ADMIN_USER_ID`。
5. 验证通过后进入 `/admin`。

### 数据访问边界

- 浏览器不能获得 `SUPABASE_SECRET_KEY`。
- 浏览器不能直接读写后台数据表和私人存储桶。
- 所有管理操作通过 Next.js Server Actions 或 Route Handlers。
- 服务端验证管理员后，才使用 Secret Key 访问数据库和 Storage。
- 公开页面也由服务端只读取并返回 `published` 内容。
- 数据库对 `anon` 和普通 `authenticated` 角色默认拒绝直接访问；第一阶段只使用简单拒绝规则，不设计复杂业务 RLS。

## 风险边界

中等方案能够明显降低后台被入侵、私人数据泄露和内容被篡改的风险，但任何公开网站都不能保证绝对安全。

公开网页必须向浏览器发送 HTML、CSS、JavaScript 和公开图片，因此无法阻止他人截图、下载公开素材或模仿视觉效果。作品归属主要通过私有 Git 仓库、提交历史、设计源文件、原始素材、域名和页面署名证明。

## 数据模型

所有可发布内容统一使用以下可见性状态：

- `draft`：草稿，仅管理员可见。
- `published`：公开发布，访客可见。
- `private`：私人内容，仅管理员可见。
- `archived`：归档内容，不在正常页面显示。

第一阶段规划的数据表：

| 表 | 职责 |
| --- | --- |
| `profiles` | 个人资料和简历入口信息 |
| `plans` | 近日规划 |
| `site_settings` | 网站基础配置和功能开关 |
| `social_links` | 社交链接及排序 |
| `home_layout` | 首页桌面布局配置 |
| `audit_logs` | 管理操作记录 |

后续扩展表：

| 表 | 职责 |
| --- | --- |
| `works` | 我的作品 |
| `collections` | 文章和视频收藏 |
| `featured_projects` | 优秀 GitHub 项目 |
| `photos` | 相册照片元数据 |

## 媒体存储

Supabase Storage 使用两个存储桶：

- `public-media`：头像、作品封面、公开相册和网站装饰图片。
- `private-media`：私人照片、未发布素材和受保护文件。

管理上传只能由验证过管理员身份的服务端执行。私人文件使用短期签名 URL 读取。上传必须限制文件类型、大小和尺寸；照片上传流程需要清除 EXIF 定位信息。重要媒体必须另行备份。

## 第一阶段开发范围

### 步骤一：认证与安全基础

- 连接 Supabase 云项目，不要求安装 Docker。
- 配置 Supabase Auth，并关闭公开注册。
- 创建单管理员登录、退出和密码恢复页面。
- 创建服务端管理员身份检查。
- 保护 `/admin` 页面及后台写操作。
- 创建简单数据库拒绝规则，禁止浏览器直接访问后台数据。
- 创建最小后台首页。
- 编写账户恢复、密钥轮换和安全检查说明。

验收条件：

- 未登录用户无法访问后台。
- 非指定管理员账户无法访问后台。
- 未登录和非管理员请求无法调用后台写操作。
- 浏览器代码、Cookie 和网络响应中不存在 Secret Key。
- 浏览器不能直接读取后台数据和私人文件。
- 登录、退出、会话过期和密码恢复流程经过真实 Supabase 云项目验证。

### 步骤二：近日规划闭环

- 规划字段：标题、说明、任务状态、优先级、进度、目标日期、相关链接、可见性状态、创建时间和更新时间。
- 支持新增、编辑、归档和移入回收站。
- 新规划默认 `private`，只有主动设置为 `published` 后才对访客可见。
- `/plans` 只展示已发布规划。
- 首页显示距离截止日期最近且已公开的未完成规划。
- 首页临期规划提醒卡片替换当前演示音乐播放器位置。

### 步骤三：网站设置与首页布局

- 基础信息与社交链接管理。
- 首页模块启用或隐藏。
- 首页桌面布局预览、拖拽、保存和恢复默认。
- 卡片吸附网格，不能拖出画布或互相遮挡。
- 核心卡片不能删除。
- 移动端使用程序定义的固定顺序。
- 保留“主题修改”入口，但第一阶段不开放颜色编辑。

## 后续阶段

- 第二阶段：我的作品、内容收藏、优秀项目。
- 第三阶段：公开和私人照片、相册管理、简历页面与 PDF。

## 备份与恢复

- 代码由私有 GitHub 仓库保存。
- Vercel 保留部署历史。
- 数据库使用 Supabase 托管备份，并提供数据导出流程。
- 重要照片和文件定期复制到独立存储位置。
- 删除内容先进入回收站，保留恢复期后再永久删除。
- 文档记录账户恢复、密钥轮换和数据恢复流程，但不记录真实秘密。

## 安装、下载与磁盘规则

- 用户 C 盘空间不足，且部分 C 盘目录可能没有 CLI 写入权限。
- 任何安装、下载、新增依赖、CLI、Docker 或大型缓存操作，执行前必须说明：
  - 安装或下载的内容。
  - 为什么需要。
  - 预计磁盘影响。
  - 默认写入位置，是否会写入 C 盘。
  - 不安装的替代方案。
- 未获得用户明确批准，不执行安装或下载。
- 优先使用项目本地缓存和 F 盘工作目录。
- 第一阶段不要求安装 Docker。

## 安全验收原则

- 默认拒绝，按需开放。
- 私人内容默认不公开。
- 后台页面和后台写操作都必须在服务端验证管理员身份。
- 任何 Secret Key 不得进入 Git、公开前端变量、浏览器存储或响应。
- 外部平台账户开启两步验证。
- 无法实际验证的安全配置，不得仅凭代码存在宣称有效。
